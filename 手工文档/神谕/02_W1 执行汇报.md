## §3 等价性报告（你点名要的第二样）

**三层证据链**（层层放宽干扰、逼近真实）：

1. **mock 控制流层**（VAD 事件脚本注入两引擎逐帧一致、决策脚本确定、绝对步频）：2 wav × 4 策略 = 8 对，**事件序列/内容/状态/轮次 8/8 逐条一致**；严格 L1 6/8，未达的 2 例即 judge=continue 配置，全部 soft 偏移 ≈0.30–0.32s = mock judge 延迟——**这是 D1.4 预注册偏差的直接指纹**（continue 超时锚从"judge 返回墙钟"改为"段末音频钟"，旧行为把 judge 延迟叠加进用户等待）。方向：Pause Handling 只会更好。
2. **真 LLM 层**（vLLM 30B-A3B，T=0/seed=42，金标 20 条）：**分类决策序列（judge/interrupt/shift 分类词+打断类型+段结构）19/20 一致**；唯一翻转在 deny 类目的 continue/switch 决策边界。L1 仅 9/20，但归因决定性证据是：**legacy 对自己都无法复现**（同 clip 两遍 legacy，3 条抽测 1 条在同一位置自分叉）——机理：legacy 墙钟 END_HOLD 使段边界帧级不可复现，±16ms 静音改变音频 base64，T=0 下 response 自由文本中途分叉（实例：两侧 response 前 18 字逐字相同后分叉）。**新引擎的段边界是确定的，旧引擎不是**——"等价性"在自由文本层的理论上限就是 legacy 自噪声地板。
3. **分数层**（HumDial 100 回归，§5）：Overall +1.75，不掉分。

附：injected 回放保真度 **L1 20/20**（对金标决策注入重放，20 条 7.5s ≈ 60× 实时）——音频钟化解锁的确定性快回放已可用作 W4 模拟器地基。

## §4 FDB-v3 审计（你点名要的第一样：Q1 答案）

**Q1（判分语义，全文最大单点风险）——风险实锤，但转化为机会：**

- `evaluate_pass_rate.py` 严格二值：名字**多重集**匹配要求 recall=1 ∧ precision=1，**任何多余调用（含同名重复）直接 FAIL**；`book→cancel→book(θ′)` 补偿序列必死（cancel 是 unexpected，第二个 book 多重集匹配后也是 unexpected）。
- 参数核对按 `actual_by_func[func].pop(0)` 对齐**首次**调用——Paris→Berlin 场景早发射 Paris 会被拿去对 Berlin 的期望参数，**双重惩罚**。
- 21 个修订场景 `state_rollback_test: true` 带 `original_param→corrected_param` 标注，`expected_tool_calls` 只含**最终意图**调用；pass-rate 报告自动按该子集分解。
- 参数判定 LLM judge 语义宽松（动态引用/别名/±5%），judge 端点可自定义（`FDB_LLM_*`）。
- **机会面**：scorer 的严格性恰好把 P2 的 C_κ 操作化——eager-launch（低延迟+脏轨迹）vs deferred-commit（高延迟+净轨迹）的权衡在官方分上可测量，**commit 停时策略成为直接优化官方 pass rate 的杠杆**；膝点实验的 y 轴之一可直接用官方分。
- 预案照旧：双轨报告（沙箱状态层终态判分器，W2 小工程——现沙箱无状态机，book_flight 永远返回 B789，需加状态记录层）+ 上游 issue/PR 草稿。

## §7 改变 W2 规划前提的五项新事实（求神谕裁断）

**7.1 guided_json 在当前 vLLM omni fork 上不可用（风险 R5 兑现）。** 实测 `guided_json`/`response_format.json_schema` 均被静默忽略（guided 与 unguided 输出逐字节相同且带 ```json fence——schema 约束下不可能出现 fence；顺带该逐字节一致性再次验证了 T=0/seed 确定性）。三条出路，倾向①：
① **prompt-only + fence 剥离解析器**：模型 T=0 下自发输出结构良好 JSON；tact offline_runner 同款解析器已在全量 100 场景验证（exact 57% 里没有一例是 JSON 解析失败造成的）；决策延迟已实测 **0.40s/次**（6s 音频入、128 token JSON 出）——蓝图§4.2 要的"guided decoding 单次延迟"用此数为上界估计。
② 决策/TTS 双实例：决策走标准 vLLM 文本实例（支持 guided + 高并发 prefix caching），TTS 走 omni 管线——与吞吐轨天然契合，**但 96GB 单卡装不下两份 30B bf16 权重**，需双卡日或串行切换。
③ 升级 omni fork 追 structured output 支持（工程风险未知）。

**7.2 决策延迟比蓝图假设低一个量级**（Blackwell 本地：分类 p50 0.10s/p95 0.12s，response 0.15/0.21s，n=99）。推论：(a) 感知冻结的绝对收益在本地部署变小——motivation 叙事应强调**结构性质**（延迟叠加不随硬件消失，云 API/边缘场景按比例放大）；(b) **Omni TTS 整句合成（p50 0.66s/p95 1.18s）成为首响延迟的最大单项**——W3 增量 TTS 的优先级是否上调？(c) P2 膝点实验的 c_w 标定应采用实测分布。
