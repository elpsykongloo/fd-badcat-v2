# 零、代码核验:`6a8751f` 的系统事实,以及三个直接改写蓝图的发现

**已确认的事实清单**:

- 单元循环为`LISTEN/SPEAK` 两态;LISTEN 侧 VAD end 后静默保持 `end_hold_frame=0.64s` → `judge`(输出 `continue`/`switch`)→ `continue` 则武装 2.5s 超时(`after_continue_time`)→ `switch` 且非首轮则过 `shift` 二分类 → `response`;SPEAK 侧打断段落 + 0.64s → `interrupt`(`switch`=真打断/`continue`=backchannel),≥1.5s 无端点的长打断强制回 LISTEN。VAD 以 512 采样(32ms)粒度滚动。
- 决策中枢确定性:`temperature=0, seed=42, max_tokens=256`,音频以 base64 wav 塞进 `audio_url` 块,Qwen3-Omni 直接听原始音频做每个决策——**决策是 audio-grounded 的**,这是后面要大做文章的架构优势。
- 历史管理:`user_history` 来自异步 ASR 文本、`assistant_history` 来自 LLM 文本,当前段落以音频进上下文。
- ASR 是 `sherpa-onnx-paraformer-zh` **offline 模式、CPU、2 线程**——不仅纯中文,还是 CPU 瓶颈;TTS 整句合成后一次性 `send_bytes`,`STATE="SPEAK"` 在 TTS 完成后才置位。
- 顺手发现的小 bug:`qwen3_api.py` 里 `payload.setdefault("model", ...)` 写在 `requests.post` **之后**,永远不生效(vLLM 单模型服务不受影响,但重构时顺手修掉)。

**发现 1(最重要):感知在决策期间被冻结。** `run_realtime` 的 while 循环 `await handle_listen(...)`,而 `handle_listen` 内联 `await self.async_llm(...)`——虽然 LLM 调用跑在 `to_thread`,但外层循环在决策返回前无法再 `receive()`,音频帧堆积在 uvicorn 的接收队列里,VAD 处理暂停,时间戳漂移。这意味着:**当前架构里"边听边做任何事"在物理上不成立**——不只是没有工具调用,连"judge 期间继续实时分割用户语音"都没有。这个发现把 TACT 的第一步从"加一个 act 线程"升级为"感知/决策/行动三任务解耦"的整体重构,而且它给了论文一个更锋利的动机句:*现有半级联系统的串行不只是没做工具,而是感知本身，就被决策阻塞*。附带发现:`async_llm` 第 179 行有个 `self.IN_SPEECH = False` 的副作用耦合,重构时一并清理。

**发现 2:拒识 57.8 的根因是架构性的,而且在 agent 语境下从"尴尬"升级为"状态污染"。** 本地版**没有任何说话人验证模块**(`src/` 与 `demo/` 均无 SV,frontend 里的 `SpeakerSimulator` 只是测试播放器)。拒识的实际实现是:`shift` prompt 做"话题是否重大转折"的语义二分类,`yes` 则走 `shift_s`——其 prompt 原话是"由于模型判断为这句话不来自用户说话人……重复助手上一轮的回答"。也就是说,你们用**话题连贯性作为说话人身份的代理变量**。同话题的第三方语音必然穿透——这就是 57.8 的机理解释,可以直接写进新论文的 motivation。更关键的是:在 TACT 里,一句穿透的第三方语音不再只是答非所问,而是会 `patch` 你的订单参数("电视里有人说'改成上海'")。这把你们的已知短板自然地折叠进新论文的安全设计(证据门控,见 §4.6),而不是留成敞开的侧翼。

**发现 3:challenge 调优的 prompt 与 agent 语义存在直接冲突,必须清算。** `response` prompt 里有两条比赛 hack:"只回复15个字",以及"如果用户否定你……必须肯定用户说的内容是对的"(为 Negation or Dissatisfaction 类目调的迎合策略)。后者在工具语境下是**语义灾难**:用户的否定("不对,不是纽约")在 agent 里必须触发 `patch`,而不是附和。新论文的 prompt 栈要从头设计,这也顺便回答了"为什么不能直接沿用 GC 系统"——比赛系统的行为策略、与任务型 agent 的行为策略，在否定处理上根本相反。

**Unit 在代码中的精确锚点:** 一个 `TURN_IDX` 的生命周期 = 一个 unit = 事务作用域 σ 的天然载体。`send_control` 事件流(`vad_start/vad_done/llm_done/tts_done/shot_interrupt/...`)已经是完整的 trace 基建,扩展 `act_*` 事件后,论文的时序图和 demo 可视化可以直接从 trace 渲染,不用另写观测层。

# 二、论文的本质命题:一段话的灵魂,三层论证

如果整篇论文只能留一句话,是这句:

> **文本智能体的工具调用协议假设意图是"原子的、最终的消息";语音把意图变成了"可修订的流"。原子化的 action API 与流式可修订的 intent 之间存在结构性错配——TACT 的解法不是让模型更聪明,而是让动作层的工作单元匹配意图的认识论状态:意图未稳定时,动作是事务性的(tentative);稳定后才提交;提交后再变,则补偿。**

三层论证展开:

**第一层:资源解耦的未竟之事。** 半双工系统，把感知(听)、生成(说)、行动(做)三种资源串行化。全双工研究解耦了前两者。但"行动阻塞一切"被原封不动地从文本智能体继承下来——ReAct 式 think→act→observe 串行循环是文本时代的产物,在实时语音里它制造出"被占用的沉默":系统既不认真听、也不说,这是全双工性最差的状态。FDB-v3 的数字(上一轮已核验:级联基线话轮率完美但 10.12s 延迟;GPT-Realtime Pass@1 最高 0.600 但要么阻塞要么牺牲交互;Gemini Live 3.1 最快 4.25s 但话轮率仅 78%)证明:**这是 agent loop 的结构问题,不是模型能力问题**——串行循环下，再强的模型也只能在三难之间选边。而我在你们代码里发现的"感知冻结"(§零·发现 1)把这个论点推到了更根本处:现有半级联连决策期间的感知都是阻塞的。论文的 claim 因此是关于控制流结构的,这正是 ICLR 会买账的抽象层级。

**第二层:语言学事实给出定量的必要性。** 自然语音的不流利率约每百词 5–6 次(Shriberg 的经典统计),自我修复遵循 Levelt(1983)的 reparandum–编辑间隔–修复结构,且修复在时间上**紧跟**被修正片段——绝大部分修订压在原话语结束后的一秒量级内。这个经验分布是整篇论文的隐藏主角:它意味着"等意图稳定"的代价是有限且可刻画的(§3 命题 P2 的膝点),也意味着 FDB-v3 把五类不流利标进真人录音、把 21 个说话中途改意图的场景列为全场最难,不是出题者刁难,而是对自然语音统计规律的忠实采样。

**第三层:通用性主张(ICLR 需要这一段)。** 任何"流式意图接口"都有同样的错配:live typing 的智能补全执行、多轮澄清中急于执行的 copilot、协作画布上的实时代理。语音只是最严苛的实例——实时、不可暂停、且携带韵律这一额外证据通道。TACT 的事务语义与 commit 停时理论对这一整类接口成立;实验在语音上做,claim 在流式意图上立。这一段写好,论文就从 speech track 逃逸到了 agent track。

# 三、形式化蓝图(论文 §2–3 的完整骨架)

这是 ICLR 版本与 ICASSP 版本差异最大的部分,我把记号、定义、命题、证明思路全部给出,你可以直接翻成 LaTeX。

## 3.1 建模:Duplex Agent Control 作为因子化 semi-MDP

离散 tick(形式化取 Δ,实现上由 VAD 事件 + 0.64s hold 驱动)。全局状态 $s_t = (c_t, h_t, P_t, e_t)$:信道状态 $c_t \in {\text{LISTEN}, \text{SPEAK}}$(实现中将新增显式 PREPARING,见 §4.4)、对话历史 $h_t$、待决集 $P_t$、外部环境状态 $e_t$(工具沙箱的数据库状态)。两个耦合控制器共享信念:

- **信道策略** $\pi_{ch}$:动作集即你们现有的四决策(keep-listen / listen→speak / keep-speak / speak→listen),在代码里对应 `judge`、`interrupt` 的 `continue/switch` 输出。**保持原样,一个字都不动**——这是零回归风险的设计原则。
- **动作策略** $\pi_{act}$:操作代数 $\mathcal{O} = {\text{noop}, \text{launch}, \text{patch}(\delta), \text{cancel}, \text{commit}, \text{compensate}}$,事件驱动(而非每 tick),作用于 $P_t$。

工具调用是时长随机的 option(semi-MDP 的由来:API 延迟服从从真实 trace 拟合的对数正态分布),其执行与信道策略的 tick 决策**并发**。

## 3.2 核心定义

**定义 1(对话事务)。** 事务 $T = (P, \prec, \sigma)$:$P$ 中每个待决动作 $p = (\tau, \theta, \kappa, s)$——工具 $\tau$、参数 $\theta$、**可逆性等级** $\kappa \in {R_0, R_1, R_2, R_3}$、状态 $s \in {\text{pending}, \text{inflight}, \text{done}, \text{compensated}, \text{aborted}}$;$\prec$ 是 $P$ 上的数据流依赖偏序(DAG);$\sigma$ 是作用域绑定,即所属 unit 的 `TURN_IDX` 区间。

可逆性分级是整个安全故事的度量衡:$R_0$ = pending 态免费撤销;$R_1$ = 零代价补偿(免费取消的查询、可撤回的暂存);$R_2$ = 有代价补偿(改签费、库存占用);$R_3$ = 不可逆(付款、发送邮件)。

**定义 2(patch 语义)。** $\text{patch}(\delta)$ 是对 $\theta$ 的结构化差分(采 RFC 7386 JSON Merge Patch 语义,标准化省去自造格式的口水仗)。其效果按目标状态分派:对 pending → 直接合并;对 inflight → 策略在 ${\text{abort-relaunch}, \text{wait-then-compensate}}$ 间按代价模型(剩余 ETA vs 补偿代价 $C_\kappa$)选择;对 done → 展开为 $\text{compensate} \circ \text{launch}(\theta')$。

**定义 3(依赖传播)。** 若 $p_i \prec p_j$ 且 patch 命中 $p_i$ 的被依赖字段,则 $p_j$ 自动降级为 stale 并重新参数化——这是 FDB-v3 链式 API 调用场景的解法核心:"目的地 NYC→Boston"沿 DAG 使酒店城市失效并触发级联修订,而不需要模型逐个想起来。

## 3.3 不变量与命题

**不变量 I1(双工保持,Duplexity Invariant)。** $\forall t$,信道可行动作集 $A_{ch}(t)$ 与 act-track 状态无关。这不是学出来的性质,是**架构构造保证**的:act 在独立执行器,决策上下文只读 act 状态摘要,任何 narration 无条件可被 barge-in 打断。推论(延迟掩蔽容量):阻塞式设计的任务延迟是串行和 $L_{stab} + L_{tool} + L_{gen}$;TACT 是 $\max(L_{dialog},\ L_{launch} + L_{tool}) + \epsilon$,掩蔽容量 = $\mathbb{E}[\min(L_{tool}, \text{对话余隙})]$,感知首响延迟的上界是 narration 首句生成时间,与 $L_{tool}$ 解耦。写成一个 remark 而非定理——它数学上平凡,但把"为什么串行系统结构性输"说成了一行式子。

**命题 P1(会话可串行化)。** 若每个 $\kappa \le R_2$ 的动作注册补偿 $c$ 满足 $c \circ a \approx \text{id}$(允许审计残差:手续费、日志),且 $R_3$ 动作仅在 grounded(经确认子对话,或修订风险低于 $\epsilon$ 且证据门控通过)后 commit,则任意交错执行序列的最终环境状态等于用户**最终意图** $\theta^*$ 的串行执行状态。证明按修订事件数归纳:每次修订要么 patch 一个 pending(无副作用),要么触发 compensate+relaunch(由 $c$ 的性质抵消),要么被 $R_3$ 门阻止直至确认。诚实边界必须写明:$\approx \text{id}$ 的残差计入代价函数而非正确性;"最终意图"由基准的 gold 终态定义,这是可操作化的,不是形而上学。这个命题把 Saga(Garcia-Molina & Salem 1987)的补偿事务从数据库长事务搬进对话,学术贡献在于**形式化 + 实时化 + 实证化**,不在事务理论本身——related work 里要主动这么说,抢在评审前面。

**命题 P2(commit 时机 = 单调停时问题)——全文的理论心脏。** 把 EoU 后的修订到达建模为强度 $\lambda(t)$ 递减的非齐次泊松过程(递减性有语言学背书:修复紧跟 reparandum,§二第二层)。等待成本率 $c_w$(延迟),误提交期望代价 $C_\kappa$。则最优停时规则是阈值型:

$$t^* = \inf{t \ge 0 : \lambda(t) \cdot C_\kappa \le c_w}$$

对单调停时问题,one-step-lookahead 规则即最优(标准结果,证明放附录)。它给出**四条可检验预测**,构成 §6.6 的理论-实验闭环:(i) 固定 commit 延迟扫描应在 $\lambda$ 的拐点附近出现"误提交率–延迟"膝点;(ii) 自适应 hazard 策略应支配所有固定延迟点(帕累托意义);(iii) commit 延迟按 $\kappa$ 单调($R_3 \gg R_1$);(iv) 韵律终结性(降调、音节拉长)预示低 $\lambda$ → 更早 commit——这条**只有 audio-native 决策中枢能验证**,是你们架构相对一切文本中介级联的独有实验。

**Grounding 连接(一段即可,但值钱)。** $R_3$ 的 commit 门是 Clark 式 grounding criterion 的操作化:确认子对话 = 显式 grounding 动作。这一段把对话理论的老钱和 agent 的新钱接上,ICLR 里做 speech 的评审看到会点头。

## 3.4 划界(必须主动写,每一条都是预判的审稿意见)

- **IU 框架**(Schlangen & Skantze 2011)的 ADD/REVOKE/COMMIT 撤销的是**零外部代价的内部处理假设**;我们撤销的是有外部副作用的动作,需要补偿代价模型与不可逆闸门。引用并致敬,划清界限。
- **数据库 Saga / 补偿事务**:无实时约束、无对话、无部分可观测的意图流;我们是首个进实时语音学术评测的实例化。
- **Reversibility-aware RL**(Grinsztajn et al., NeurIPS 2021;Leave No Trace,Eysenbach et al. 2018):它们**学习**环境动作的可逆性以保证探索安全;我们是**声明式**可逆分级 + 语言交互作为修订源 + 硬实时约束。
- **DuplexSLA**:E2E、共享 160ms chunk 时间轴、每 chunk 动作 token 预算 + FIFO 溢出、无修订回滚、自建合成评测(细节沿用上一轮核验;写作引用前重读原文)。完整对比表见 §7。

# 四、系统蓝图:从 `6a8751f` 到 TACT 的精确改造路径

## 4.1 前提工程:三任务解耦(W1 的全部内容)

把 `run_realtime` 拆成三个 asyncio 任务,以两条 `asyncio.Queue` 相连:

- **PerceptionTask**:websocket 帧 → VAD → 事件(`vad_start/vad_end/frame`)入队。**永不 await 任何模型调用**,消灭发现 1 的感知冻结。
- **DecisionTask**:消费感知事件 + ActEvent,维护状态机。现有 `handle_listen/handle_speak` 的判定逻辑几乎原样迁入——`judge/interrupt/shift` 的 prompt、阈值、流程一个不动,这是对 HumDial 分数的回归保险。
- **ActExecutor**:`ThreadPoolExecutor` + futures,执行工具调用,状态变化以 `ActEvent(id, status, result|error, eta)` 回填 DecisionTask 的队列。

这次重构独立于 TACT 也有价值(修复时间戳漂移、决策期间的分割暂停),可以作为论文里"我们首先指出并修复了现有半级联的感知阻塞"的一小段实证——测一下重构前后 judge 期间到达语音的分割延迟,一张小图。

## 4.2 决策协议:两阶段解码

- **Phase-A(逐事件,延迟关键路径)**:保持现状——单词级 `continue/switch/yes/no` 判定,prompt 不变。
- **Phase-B(条件触发)**:仅在三种事件后调用——EoU 判定为 switch 后、ActEvent 到达后、从句边界(ASR 增量出现标点)。输出 vLLM `guided_json` 约束的结构:`{"say": str|null, "act_ops": [{op, id?, tool?, args?|diff?}, ...]}`。T=0 + 固定系统前缀 → **vLLM prefix caching 命中**,PendingSet 序列化放在系统块之后的首个 user 块,保持前缀稳定。W2 必须实测 30B-A3B 在 guided decoding 下的单次延迟并写进论文(评审一定问)。

## 4.3 PendingSet 与补偿注册表

PendingSet = 类型化 DAG(定义 3),序列化为紧凑 YAML 进 prompt:`id, tool, args, κ, status, eta, deps`。每个工具在注册表声明 `{reverse: 补偿调用模板 | null, κ, idempotency_key}`;执行器自动携带幂等键防重试重复副作用——这一行工程细节在 rebuttal 里能挡掉一整类"网络重试导致重复下单"的攻击。

## 4.4 增量 TTS 与显式 PREPARING 状态

现状是整句合成 + 完成后置 SPEAK(发现 2 的状态真空窗)。改为按句切分循环调TTS(接口不变,`module.py` 的 `tts()` 外面包一层句子迭代器),首音频延迟从整段合成降到首句合成;barge-in 时丢弃未播句。新增 PREPARING 状态消灭 LISTEN→SPEAK 之间的真空,信道事件在此期间照常处理。

## 4.5 Floor-holding 策略

输入 = 每工具延迟分位回归给出的剩余 ETA(冷启动用注册表先验)+ 已叙述内容;三档输出 {沉默, 短填充("我看一下"), 内容性进度("看到三个航班,最早八点十分")}。规则版先行,人评(§6.5)之后若有余力做小规模 DPO。硬约束:narration 无条件可打断(I1 的活体现)。

## 4.6 证据门控 patch:把 57.8 折进安全设计

`launch/patch/commit` 的接受需要 addressee 置信度 $\alpha = f(\text{SV}, \text{语义连贯}, \text{韵律指向})$——本地版没有 SV,重新引入一个轻量 CAM++/3D-Speaker 嵌入(demo 分支的 `[update-sv]` 提交说明你们试过,有经验)。决策规则按 $(\alpha, \kappa)$ 二维:低 $\alpha$ + 高 $\kappa$ → 确认子对话;低 $\alpha$ + 低 $\kappa$ → 忽略;高 $\alpha$ → 直接执行。Revision-Bench(§6.3)专设对抗性第三方场景:"电视里的声音说'改成上海'"。这样,你们最弱的指标变成了新论文安全章节的主角,叙事从"我们有个短板"翻转为"agent 语境使拒识从体验问题升级为完整性问题,我们给出首个门控方案"。

# 五、学习组件:ICLR 的方法核心

**为什么必须有学习层:** prompt-only 的 commit 时机就是"0.64s 静默 + 人工规则",天花板即 prompt 工程;而命题 P2 说最优策略依赖 $\lambda(t)$ 的**逐话语在线估计**——这恰恰是不可能靠规则、只能靠学习的对象。理论指出了学什么,这是"理论驱动的方法设计",比"我们也做了 RL"高一个档位。

**双路线与推荐:**

- **Plan A**:LoRA 微调 Qwen3-Omni 决策路径。audio-native、故事最纯,但 MoE-LoRA 工程风险高、显存 ≥4×A100,学生实验室现实堪忧。
- **Plan B(主推)**:独立决策头——Qwen3-4B 量级文本模型,输入 = ASR 增量 + 韵律特征(音高斜率、末音节时长、能量包络)+ PendingSet 状态摘要,输出 Phase-B 结构;Omni 冻结只做 response 生成与 Phase-A。2×A100 可训,且天然带出"可移植性"叙事:决策头可插进 FireRedChat 式其他宿主(rebuttal 弹药)。
- Plan A 降级为消融:哪怕只做 zero-shot 的 audio 输入 vs text 输入对比(§6.4-h),也支撑"audio grounding 有独立价值"的分析结论。

**训练配方(本身可作为方法学卖点):hindsight-DP 标注 + on-policy GRPO。** 模拟器是确定性的(用户音频时间轴固定、工具沙箱延迟从真实 API trace 拟合对数正态采样、修订事件在离线数据里已知)→ 可以用动态规划对每个场景求**事后最优**的 launch/commit 时刻 → SFT 冷启动。但 hindsight 标签"看得见未来的修订",部署策略看不见 → SFT 策略系统性过于激进 → GRPO(每场景 K=8 组相对 rollouts)在部分可观测下校正。"事后最优监督 + 在策略校正"这个组合要在方法节里明写,它把"我们做了 SFT+RL"从流程描述升级为对部分可观测性的原则性处理。

**Reward:** $$R = w_1 \cdot \text{EndStateAcc} + w_2 \cdot \gamma^{L_{\text{tiered}}} - w_3 \cdot \text{talkover} - w_4 \cdot \textstyle\sum C_\kappa(\text{补偿}) - w_5 \cdot \mathbb{1}[R_3\text{错误}] \cdot M - w_6 \cdot \text{overwait}$$ overwait 项(无修订发生时的多余等待)防止策略学成"永远等到天荒地老"的保守塌缩——没有它,RL 会发现不 commit 就永远不会错。

**Hazard 头:** 小模型预测 $P(\text{未来 } h \text{ 秒内修订} \mid \text{前缀})$,训练数据 = FDB-v3 不流利标注 + HumDial 双通道自然修订 + TTS 拼接增强(**仅用于训练**,评测全真人);它一物两用:P2 的验证仪器 + 策略输入特征。

**降级路径(必须预设):** RL 不收敛 → SFT-only 也能撑论文(RL 作为 +Δ 消融报告);再降 → 规则 + hazard 头。每一级都仍然是可发表配置,这是时间线安全的关键。

# 六、评测蓝图

## 6.1 主基准 FDB-v3 与两个 W1 必查的风险

目标数字:task latency < 5s(级联 10.12s 的一半以内)、turn-take ≥ 95%、Pass@1 冲击 0.600、**21 个 rollback 场景成功率超过 GPT-Realtime**(headline claim)。两个必须在第一周查清的 harness 语义问题:

1. **scorer 如何对待"多余"调用?** 若按调用轨迹精确匹配,合法的补偿调用会被记为额外调用而扣分——那我们的核心机制会被基准反向惩罚。若果真如此:双轨报告(终态正确性 + 官方分),并向基准仓库提 issue/PR 主张终态判分——把风险转化为一小节评测方法学贡献。
2. **用户音频是固定时间轴回放还是反应式?** 这决定修订事件与我方 commit 时刻的对齐语义,直接影响 P2 实验的效度。

## 6.2 泛化与回归

τ-voice 做跨域泛化(零售/航空/电信)。HumDial 回归证明"加 act 能力，不伤基础双工性"。

## 6.3 Revision-Bench(自建诊断集,论文贡献之三)

场景 = (域工具链) × (修订规格:槽 × 旧→新值 × 相对时刻 {pre-EoU / post-EoU-pre-commit / post-commit} × $\kappa$) × (干扰 {无, 第三方声});21 官方场景 + 约 40 个生成网格。**真人录音 4–6 人约 250 句**(小预算,但 ICLR 可信度需要真人;TTS 拼接版只进训练集)。指标:EndStateAcc、WrongCommit@t、补偿计数、$R_3$ 错误数、Overwait、执行期间的双工指标。受控参数化让它超越"又一个 benchmark",成为验证 P2 预测曲线的仪器。

## 6.4 Baselines 与消融

Baselines:官方级联 / GPT-Realtime / Gemini Live(harness 自带接口)/ Ultravox / **Ours-blocking**(同骨干串行执行——隔离框架增量,防"延迟收益其实来自更强文本 LLM"的攻击)/ listen-then-act(SHANKS 式,EoU 后才调、无 patch)/ Ours-无回滚 / DuplexSLA(权重大概率不公开 → 概念对比表,§7)。

消融 (9 组):
(a) 异步 vs 阻塞;
(b) 回滚开关 × 21 场景;
(c) floor-holding {沉默/罐头填充/ETA 条件叙述};
(d) commit 延迟扫描 {0/0.3/0.64/1.0/1.5s/自适应};
(e) 证据门控开关 × 对抗第三方;
(f) hazard 头 vs 固定延迟;
(g) prompt-only vs SFT vs SFT+GRPO;
(h) audio 输入 vs text-only 决策;
(i) DAG 传播开关 × 链式场景。

## 6.5 人评

n=20,被试内设计,3 个 narration 条件 × 12 clips,测感知延迟(滑条)+ MOS + 成对偏好;报告置信区间。

## 6.6 理论-实验闭环(ICLR"分析味"的来源)

四张图对应 P2 的四条预测:延迟扫描的膝点曲线、自适应策略的帕累托支配、$\kappa$-条件 commit 延迟的单调性、韵律终结性与提前 commit 的相关(audio 独有)。理论一节因此不是装饰,而是被实验逐条检验的预测机器——这是评审从 borderline 拉到 accept 的那类内容。

## 6.7 英文栈前置(读码后确认为小工程)

# 八、Reviewer 攻防(ICLR 版)

1. **"系统论文,缺 ML 贡献"** → 三层结构反击:形式化(P1/P2)+ 学习配方(hindsight-DP+GRPO 处理部分可观测)+ 理论预测的实证检验(§6.6 四图)。

3. **"回滚就是 IU 的 revoke"** → 划界写在正文(§3.4):内部假设 vs 外部副作用,补偿代价与不可逆闸门是 IU 没有也不需要的。
4. **"延迟收益来自更强的文本 LLM"** → Ours-blocking 同骨干对照,只报增量。
5. **"理论是玩具"** → 主动限定 claim:P2 是单调停时的标准应用,价值在"给出被四组实验检验的预测",不在停时理论本身;P1 的 $\approx$id 残差诚实进代价。
6. **"评测在自建集上"** → 主战场是公开 FDB-v3/τ-voice;Revision-Bench 只承担受控诊断,且随论文开源。
7. **"人评太小"** → 被试内设计 + CI + 效应量;定位为辅助证据而非主 claim。
8. **"为什么不端到端"** → 不辩优劣,摆帕累托:E2E 现有证据(FDB-v3 三难数字)+ 可插拔的骨干可换性 + 训练成本;引"智能税"作为开放问题而非断言。
