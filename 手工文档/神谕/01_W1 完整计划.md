# Day 1(7/6 周一):事件模型落地 + FDB-v3 定位 + 行政项启动

**D1.1 事件类型与队列骨架(上午)。** 新建 `src/engine.py`:

```python
@dataclass
class FrameEvent:    seq: int; t_audio: float; t_wall: float; pcm: np.ndarray
@dataclass
class DecisionDone:  kind: str        # "judge"|"shift"|"interrupt"|"response"|"shift_re"
                     epoch: int; turn: int; text: str; infer: float
@dataclass
class TtsDone:       turn: int; wav_path: str; dur_audio: float
@dataclass
class ControlMsg:    kind: str; data: dict   # "session_end" 等;W3 起 act_* 事件复用此通道
```

单队列 `self.q: asyncio.Queue`,类型分派。`t_audio = seq * 256 / 16000`(与客户端帧格式一致,由 reader 计数)。

**D1.2 reader 协程(下午)。** 全部职责三行:`receive()` → 构造事件 → `put_nowait`。永不 await 模型、永不碰引擎状态。文本控制消息(`event:"end"`)同样走队列,由引擎串行处理 reset——消灭 reset 与在途决策之间的另一处潜在竞态(旧代码里 `reset()` 在 receive 循环里直接调,若与在途 LLM 交错,历史列表会被并发读写)。

**D1.3 引擎协程主循环(下午)。**

```python
async def engine_loop(self):
    while True:
        ev = await self.q.get()
        if   isinstance(ev, FrameEvent):   self._on_frame(ev)      # VAD + 状态机
        elif isinstance(ev, DecisionDone): self._on_decision(ev)   # 陈旧性检查后分派
        elif isinstance(ev, TtsDone):      self._on_tts(ev)
        elif isinstance(ev, ControlMsg):   self._on_control(ev)
```

今天先做"事件化但决策仍内联"的中间版:`_on_frame` 里迁入 `handle_listen/handle_speak` 的逻辑但暂时保留 `await async_llm`——目的是先验证 **reader/engine 分离 + 音频钟** 两件事的等价性,把决策分叉留给 Day 3,一次只引入一个变量。

**D1.4 音频钟迁移(与 D1.3 同步做)。** 逐变量替换表——每一行都对应你验证里点名的漂移点:

# Day 3(7/8 周三):决策分叉 + 陈旧性协议——本周技术核心

**D3.1 决策分叉。** 四个调用点(judge/shift/interrupt/response)统一改为:

```python
def dispatch_decision(self, kind, prompt, audio, *, epoch, turn):
    async def _run():
        text, infer = await asyncio.to_thread(llm_call, ...)
        self.q.put_nowait(DecisionDone(kind, epoch, turn, text, infer))
    self.inflight[kind] = asyncio.create_task(_run())
```

引擎发出 dispatch 后**立即返回事件循环**——感知永不停摆。决策结果作为事件回来,过陈旧性检查后才作用于状态机。`self.IN_SPEECH` 等一切状态改动从 `async_llm` 内部移出(:178 的副作用正式清除),收归 `_on_decision`。

**D3.2 陈旧性协议(整个重构里最需要想清楚的 20 行)。** 引入 `seg_epoch`:每当"判定所依据的音频段"发生增长或作废时自增——具体触发:LISTEN 态 judge 在途期间出现新 `vad_start`;打断段在 interrupt 在途期间续上新语音;`reset`;turn 切换。判定表:

| 在途决策     | 期间事件             | 结果到达时处理                                               |
| ------------ | -------------------- | ------------------------------------------------------------ |
| judge(e)     | 无(epoch 不变)       | 正常执行 continue/switch 分支                                |
| judge(e)     | vad_start(epoch→e+1) | 结果为 `continue`:与现实一致,丢弃即可(用户确实在继续);结果为 `switch`:**作废**,等下一次 EoU 用增长后的完整 BUFFER 重新 judge |
| shift(e)     | vad_start            | 同上作废,重走 judge→shift 链                                 |
| interrupt(e) | 打断段续上新语音     | 作废,按扩展后的段重判(等价于旧系统"帧堆积后重新累积"的意图,但不再有时间戳污染) |
| response(e)  | 用户开口             | **W1 保持遗留语义**:response 与 TTS 照常完成,用户语音由感知正常分割、进入当前态的常规处理(即 SPEAK 态后走 interrupt 流)。不引入"取消在途 response"——那是 W3 PREPARING 状态 + 增量 TTS 的事,flag 预留:`engine.cancellable_response: false` |

关键设计判断说明:作废 `switch` 而保留 `continue` 不是随意的——`continue` 的语义是"不动作",与任何后续现实都相容;`switch` 的语义是"开始回应",而它的证据(用户说完了)已被新语音证伪。这个不对称性值得在论文的 system 节写一句,它是"决策与感知并发后如何保持一致性"的最小完整答案。

**D3.3 SPEAK→LISTEN 的 playback 时钟(flag 默认关)。** 你我都注意到，旧系统没有"播放结束→回 LISTEN"的转换,下一轮用户语音永远走 interrupt 分类器——这在 HumDial 上是既定行为,**不动**。但 agent 模式需要知道"我说完了没"(floor-holding 的前提),所以今天顺手加:`TtsDone` 携带 `dur_audio`,引擎在音频钟上推进一个 `playback_end = t_sent + dur` 的估计;`engine.playback_autoend: false` 时只记录不转移(HumDial 模式),`true` 时到点自动回 LISTEN(agent 模式,W3 起用)。零风险,两种模式共存。

**D3.4 感知冻结的 before/after 测量(论文图 1 候选)。** 指标:**分割处理延迟** = `t_wall(引擎处理某 VAD 事件) − t_wall(该事件对应音频末帧到达 reader)`。方法:金标集里那 3 条边界音频 + 1 条构造音频("用户长独白,中途每逢系统 judge 窗口就插入短语音"),旧引擎 vs 新引擎各跑 5 遍,画 p50/p95/p99。预期:旧 = 决策延迟量级的尖峰(单次 judge 数百 ms 至秒级,三连调用最坏叠加),新 < 10ms 恒定。同时报队列深度峰值(旧:~62 帧/秒的积压;新:≤2)。这张图 + 你验证里的机理描述 = 论文 motivation 一节的完整实证。

**Day 3 验收:** 陈旧性协议 8 个单测全绿(纯事件驱动,scripted mock LLM 带可控延迟);金标集 L1/L2 等价复验通过;before/after 图初版产出。

## S3 config.yaml 增量(全部默认值 = 复现旧行为)

```yaml
mode: humdial            # humdial | agent_blocking | agent   (agent=W2起)
engine:
  arch: actor            # actor | legacy(backend_legacy.py)
  playback_autoend: false
  cancellable_response: false
llm:
  audio_block: audio_url # audio_url | input_audio(云端兼容)
  decision_timeout_s: 15 # 新增:单次决策硬超时,超时按保守动作回退(judge→continue),
                         # 替代旧的 300s——一次卡死不再冻结整轮对话五分钟
asr:
  backend: paraformer_zh # paraformer_zh | sensevoice | zipformer_bi
  provider: cuda
prompts_en: {...}        # Day 4 建骨架
```

`decision_timeout_s` 的回退动作要在表里定死:judge→`continue`(保守:多等不误判);interrupt→`continue`(保守:不轻易让步);shift→`no`;response 超时→重试一次后播放兜底致歉句。这是新引入的行为,记入已知偏差栏,HumDial 回归里验证其从不触发(正常延迟远低于 15s)。
