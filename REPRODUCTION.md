# fd-badcat Reproduction Notes

This checkout is prepared for AutoDL with separate conda environments and `uv`
for Python package management. Qwen3-Omni is served with `vllm-omni`, not the
original custom vLLM fork.

## 1. System tools

```bash
apt-get update
apt-get install -y aria2 tmux
python -m pip install --upgrade uv
```

AutoDL academic acceleration is enabled automatically by setup scripts when
`/etc/network_turbo` exists. Disable it with:

```bash
USE_NETWORK_TURBO=0 bash setup/download_assets.sh qwen
```

The startup scripts set `NO_PROXY/no_proxy` for `127.0.0.1`, `localhost`, and
`0.0.0.0`. Keep this when network acceleration is enabled; otherwise local
requests to ports `10003`, `10004`, `19000`, and `18000` can be routed through
the local proxy and hang.

## 2. Conda environments

```bash
bash setup/qwen3omni_env.sh
bash setup/indextts_env.sh
bash setup/aux_model.sh
```

fdbc-qwen3o-vllm   # Qwen3-Omni vllm-omni service, Python 3.12

## 3. Assets

High-concurrency Hugging Face downloads use `aria2c`. Qwen weights default to
one large file at a time with 16 aria2 segments to avoid Hugging Face 429
throttling while keeping segmented download enabled:

```bash
bash setup/download_assets.sh qwen data asr
```

Optional Index-TTS checkpoint download uses the original ModelScope source:

```bash
bash setup/download_assets.sh index-tts
```

Expected asset layout:

```text
model/
├── Qwen3-Omni-30B-A3B-Instruct/
├── index-tts-vllm/
│   └── checkpoints/
│       └── Index-TTS-1.5-vLLM/
└── sherpa-onnx-paraformer-zh-2024-03-09/
data/
└── HumDial-FDBench/
```

Qwen3-Omni is about 70 GB. Keep `HF_HOME` and all downloads on the workspace
disk to avoid duplicate cache copies.

Extract the HumDial test archive before running real samples:

```bash
bash setup/extract_humdial.sh
```

The extracted test set is expected at:

```text
data/HumDial-FDBench/extracted/test/
├── cn_test_nondev/
└── en_test_nondev/
```

## 4. Startup

Terminal 1:

```bash
bash setup/start_qwen3omni_vllm_omni.sh
```

The default Qwen deploy config is `configs/qwen3_omni_text_only.yaml`, which
uses vllm-omni but loads only the Qwen3-Omni thinker/text stage. This matches
fd-badcat because final speech is generated by Index-TTS. Override
`QWEN_DEPLOY_CONFIG` if you want a full Qwen3-Omni audio-output pipeline.
`setup/start_qwen3omni_vllm_omni.sh` also applies
`scripts/patch_prometheus_instrumentator.py`; this avoids a
`'_IncludedRouter' object has no attribute 'path'` metrics middleware failure
seen with the current vllm-omni/FastAPI stack.

Terminal 2:

```bash
bash setup/start_qwen3_proxy.sh
```

Terminal 3:

```bash
bash setup/start_index_tts.sh
```

On RTX PRO 6000 Blackwell, Index-TTS defaults to
`INDEX_TTS_ENFORCE_EAGER=1`. The setup/start scripts apply
`scripts/patch_index_tts_vllm.py`, which makes eager mode configurable and adds
a `soundfile` fallback for reference WAV loading when torchaudio expects
`torchcodec`.

Terminal 4:

```bash
bash setup/sc.sh
```

`setup/sc.sh` starts backend and frontend in a `tmux` session. If you prefer
manual startup:

```bash
conda activate fd-sds
python src/backend.py --config src/config.yaml
python src/frontend.py --config src/config.yaml
```

## 6. Smoke tests

After the three model services are running:

```bash
curl --noproxy '*' http://127.0.0.1:10003/v1/models
curl --noproxy '*' http://127.0.0.1:19000/health

conda activate fd-sds
python -c "from pathlib import Path; from src.module import tts; tts('你好，模块测试成功。', Path('logs/module_tts_smoke.wav'))"
python scripts/smoke_backend_ws.py \
  --config src/config.yaml \
  --input logs/module_tts_smoke.wav \
  --output logs/backend_ws_tts.wav
```

The websocket smoke should emit `vad_start`, `vad_done`, `vad_640_done`,
two `llm_done` events, `tts_done`, and save a 16 kHz WAV returned by the backend.

To run a real HumDial small-sample smoke test, keep Qwen, Qwen proxy,
Index-TTS, and the backend running, then execute:

```bash
bash scripts/run_humdial_smoke.sh
```

This uses two real files from the extracted HumDial test set:

```text
data/HumDial-FDBench/extracted/test/cn_test_nondev/pause/0009_0002.wav
data/HumDial-FDBench/extracted/test/en_test_nondev/pause/0006_0052.wav
```

Expected outputs:

```text
logs/humdial_smoke/cn_pause_0009_0002_output.wav
logs/humdial_smoke/en_pause_0006_0052_output.wav
```

For a larger seeded batch, the runner samples evenly across language/category
folders, sends each full HumDial WAV through the backend, and writes one mixed
assistant-output WAV plus event logs per sample:

```bash
conda run -n fd-sds python scripts/run_humdial_batch.py \
  --count 100 \
  --seed 42 \
  --out-dir logs/humdial_100 \
  --resume
```

Batch artifacts:

```text
logs/humdial_100/manifest.jsonl
logs/humdial_100/summary.csv
logs/humdial_100/samples/*/output.wav
logs/humdial_100/samples/*/events.jsonl
```

## 7. Strict Evaluation Setup

The original evaluation stack uses FunASR for Chinese/mixed transcription,
NeMo Parakeet for English transcription, and DeepSeek chat as the judge.

Install evaluation dependencies into the existing GPU runtime environment:

```bash
bash setup/eval_env.sh
```

Download/preload the original transcription models:

```bash
bash setup/download_eval_models.sh
```

Before running judge-based evaluation, source the environment template and set
your DeepSeek key:

```bash
source configs/eval.env.example
export DEEPSEEK_API_KEY=sk-...
```

Defaults keep the original model choices:

```text
FunASR: paraformer-zh@v2.0.4 + fsmn-vad@v2.0.4
NeMo:   nvidia/parakeet-tdt-0.6b-v2
Judge:  deepseek-chat at https://api.deepseek.com
```
